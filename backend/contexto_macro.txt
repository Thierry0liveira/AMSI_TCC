## QUANDO O FRONTEND PEDE ALGO NOVO

Perguntas a responder antes de codar:

  1. Isso já existe? Verificar se a rota ou lógica já está implementada.
     Consultar /docs (Swagger) com o servidor rodando.

  2. É uma variação de algo que existe ou é algo novo?
     - Variação (ex: filtro em listagem existente): adicionar parâmetros na
       rota existente, não criar rota nova.
     - Algo novo (ex: relatório, agregação): criar rota nova seguindo o padrão.

  3. Exige mudança no banco? Se sim:
     - Atualizar tabelas_do_banco.txt ANTES de codar
     - Atualizar o model correspondente
     - Atualizar o schema correspondente
     - Só então criar/modificar a route

  4. Quem pode acessar? Definir se é get_current_user ou exige_admin
     antes de implementar.

## PADRÃO DE ROTA NOVA

Estrutura de arquivos a seguir:
  models/     — define a tabela (SQLAlchemy)
  schemas/    — define o que entra e o que sai (Pydantic)
  routes/     — define os endpoints (FastAPI)

Nunca pular etapas. Sempre model → schema → route.
Registrar a nova route no main.py.

Nomenclatura:
  - Prefixo da rota: plural, minúsculo, sem acento (ex: /lancamentos/, /usuarios/)
  - Funções: verbo + substantivo (ex: listar_lancamentos, criar_clifor)
  - Models: PascalCase (ex: Lancamento, ClienteFornecedor)
  - Schemas: PascalCase + sufixo Create/Update/Response (ex: LancamentoCreate)

## PADRÃO DE FILTROS EM LISTAGEM

Query parameters opcionais na rota GET. Todos opcionais, sem filtro = retorna tudo.
Filtros se acumulam (AND implícito). Nunca criar rota separada só para filtrar.

## PADRÃO DE RESPOSTA DE ERRO

  404 — recurso não encontrado
  400 — dado inválido enviado pelo cliente
  401 — não autenticado
  403 — autenticado mas sem permissão
  409 — conflito (ex: email já cadastrado)
  500 — erro interno (deixar o FastAPI tratar automaticamente)

Nunca retornar 200 com mensagem de erro no body.

## PADRÃO DE COMUNICAÇÃO FRONTEND ↔ BACKEND

O frontend sempre:
  - Manda Authorization: Bearer <token> em todas as requisições autenticadas
  - Recebe JSON
  - Trata 401 redirecionando para tela de login
  - Trata 403 mostrando mensagem de permissão insuficiente

O backend sempre:
  - Aceita e retorna JSON
  - Nunca retorna o campo `senha` em nenhuma resposta
  - Datas em ISO 8601 (ex: "2026-04-19T10:00:00")
  - Decimais como string no JSON (comportamento padrão do Pydantic com DECIMAL)

## ADICIONANDO CAMPO NOVO NO BANCO

Checklist:
  1. ALTER TABLE no pgAdmin
  2. Atualizar tabelas_do_banco.txt
  3. Atualizar model (models/)
  4. Atualizar schemas/ (Create, Update e Response conforme necessário)
  5. Testar com a bateria ou manualmente no Swagger
  6. Atualizar fase_atual.txt se for pendência resolvida

## LÓGICA DE NEGÓCIO — INADIMPLÊNCIA

  - Clifor é inadimplente se tiver lançamento de Crédito: vencido + não pago + não estornado
  - Gerenciado em dois lugares (redundância intencional):
    1. Trigger trigger_inadimplente no banco (INSERT/UPDATE/DELETE em lancamento)
    2. utils/inadimplencia.py → atualizar_inadimplente(id_clifor, db)
       Chamado em: POST /lancamento/, PUT /lancamento/{id}, DELETE /lancamento/{id}
